# Master Grading Standard

Use this across all assignments. It operationalizes the **six non-negotiable
modeling standards** in `CURRICULUM.md` and is applied through the **four review
passes** in `.agent_private/REVIEW_PROTOCOL.md`.

## The six non-negotiable standards (grading foundation)

A submission cannot Pass if any of these is violated:

1. Every forecast row has a forecast **origin and target date**.
2. Every feature has a **unit, source, transformation, and information-availability**
   assumption.
3. Every model is compared against **at least one simple baseline**.
4. **No random train/test split** for time-series forecasting (unless the
   assignment explicitly asks for a contrast).
5. **No leakage** — never evaluate on data used to compute the feature or benchmark.
6. The report states **what would make the model fail** and never claims causality
   from correlation alone.

Each section below is one of the four review passes; the checks within it are how
the six standards are verified.

## 1. Reproducibility (pass 1)

- Script runs from repo root.
- Outputs are regenerated by command.
- No raw data modifications.
- No hard-coded machine-specific absolute paths.
- API keys are not committed.

## 2. Data correctness (pass 2 — enforces standard 2)

- Dates are parsed and sorted correctly.
- Units are checked and converted if needed.
- Frequency alignment is explicit.
- Missingness is measured and handled deliberately.
- Every feature has a unit, source, transformation, and information-availability
  assumption.

## 3. Modeling correctness (pass 3 — enforces standards 1, 3, 4, 5)

- Target, horizon, and forecast origin are explicit (every row has origin + target
  date).
- At least one simple baseline is included and the model is compared to it.
- Validation is time-ordered; no random split.
- No target leakage — nothing used in evaluation was used to build the feature or
  benchmark.
- Metrics match the objective.

## 4. Interpretation (pass 4 — enforces standard 6)

- Report ties results to gas-market fundamentals when relevant.
- Limitations are specific, including an explicit statement of what would make the
  model fail.
- Claims are calibrated to evidence; no causality claimed from correlation alone.
- Next questions are concrete.

## 5. Code quality (supporting)

- Functions are small.
- Variable names are meaningful.
- Intermediate outputs can be inspected.
- Plots and tables are saved to `outputs/`.
